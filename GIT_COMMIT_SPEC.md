# Git Commit 规范

## 规范概述

Git Commit 消息应分为三个部分：标题、内容列表和小结。这种格式有助于清晰地传达提交的目的和变更内容，便于团队协作和代码审查。

## 规范格式

```
<type>: <title>

<content-list>

<summary>
```

### 1. 标题部分

- **格式**：`<type>: <title>`
- **类型（type）**：使用动词的过去式或现在分词，描述变更的性质
  - `feat`：新增功能
  - `fix`：修复bug
  - `refactor`：代码重构，不涉及功能变更
  - `docs`：文档更新
  - `style`：代码格式调整（不影响功能）
  - `test`：测试用例更新
  - `chore`：构建流程或辅助工具的变更
- **标题（title）**：简洁明了，不超过50个字符
- **首字母大写**：标题首字母大写
- **结尾无标点**：标题结尾不使用句号或其他标点符号

### 2. 内容列表部分

- **格式**：使用项目符号（`-` 或 `*`）列出具体变更内容
- **条目清晰**：每个条目描述一个具体的变更点
- **简洁明了**：每个条目不超过72个字符
- **使用祈使句**：以动词开头，描述具体动作
- **空行分隔**：与标题和小结之间用空行分隔

### 3. 小结部分

- **格式**：简要总结变更的影响或结果
- **简洁明了**：不超过72个字符
- **空行分隔**：与内容列表之间用空行分隔

## 示例

```
feat: 实现周期待办事项功能

- 支持多种周期类型：年/月/周/日/小时/分钟
- 每周周期支持特定日期选择
- 标记完成后自动更新到下一次出现时间
- 完善了README文档，添加了周期功能说明
- 更新了数据库架构，添加了周期相关字段
- 实现了自动数据库迁移
- 添加了完整的端到端测试覆盖

测试结果：全部13个测试用例通过
```

## 最佳实践

1. **一个提交一个功能**：每个提交应只包含一个功能或修复，便于代码审查和回滚
2. **提交前检查**：确保提交的代码符合项目的代码规范和测试要求
3. **使用语义化类型**：选择合适的类型，清晰描述变更的性质
4. **避免大型提交**：大型提交应拆分为多个小型提交，便于理解和审查
5. **关联Issue**：如果提交与某个Issue相关，应在提交信息中关联该Issue

## 工具支持

- **git commit-msg钩子**：可以使用git钩子来验证提交信息是否符合规范
- **commitlint**：可以使用commitlint工具来强制实施提交规范
- **IDE插件**：许多IDE都有Git Commit模板插件，可以帮助生成符合规范的提交信息

## 总结

遵循统一的Git Commit规范有助于提高代码的可维护性和团队协作效率。通过清晰的标题、详细的内容列表和简洁的小结，可以让团队成员快速理解提交的目的和变更内容，便于代码审查和问题定位。
